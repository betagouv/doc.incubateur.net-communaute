# Guide pour coder avec l'IA

La doc [Faire de l'IA chez beta](https://doc.incubateur.net/communaute/gerer-son-produit/faire-de-lia-chez-beta) permet de se poser les bonnes questions avant d'implémenter des features IA dans son application.

Le présent doc couvre l'usage d'IA générative pour le code ou dans des workflows de traitement (ex: n8n).

### Bonnes pratiques

Ne **jamais** travailler avec des données ou credentials de PROD en local.

Interdire à son agent l'accès aux fichiers sensibles. (exemple avec [Claude code](https://code.claude.com/docs/en/settings))

Lancer les agents dans des VMS, par ex avec [docker sandox](https://www.docker.com/products/docker-sandboxes/) ou [agent-vm](https://github.com/sylvinus/agent-vm).

Utilisez (et contribuez) aux skills officiels proposés par l'équipe IAE : [etalab-ia/skills](https://github.com/etalab-ia/skills).

### Coût écologique

Utiliser un LLM peut être coûteux et sa pertinence doit être évaluée.

Voir rapport ARCEP Mai 2026 : [intelligence artificielle generative : quels defis environnementaux ?](https://www.arcep.fr/uploads/tx_gspublication/rapport-IA-et-environnement_210526.pdf)

### Modèles recommandés

Si vous traitez des données sensibles, rapprochez vous de [l'équipe Albert sur Tchap](https://www.tchap.gouv.fr/#/room/#albert:agent.dinum.tchap.gouv.fr) qui pourra vous fournir des accès à des endpoints LLMs dans des environnements SecNumCloud. 👉 [voir les modèles](https://albert.sites.beta.gouv.fr/solutions/models/).

L'équipe animation peut également vous fournir des tokens API pour les endpoints Scaleway. 👉 [voir les modèles](https://www.scaleway.com/en/docs/generative-apis/reference-content/supported-models/) ("We do not collect, read, reuse, or analyze the content of your inputs, prompts, or outputs generated by the API". [source](https://www.scaleway.com/en/docs/generative-apis/reference-content/data-privacy/))

### Juridique

* Le no code ne constitue pas une catégorie juridique autonome : il est pleinement soumis au régime des systèmes d’information de l’État, indépendamment du mode de développement. Il s’agit du principe de neutralité technologique que l’on retrouve au sein du [Décret n°2019-1088 du 25 octobre 2019](https://https/www.legifrance.gouv.fr/loda/id/JORFTEXT000039281619).
* Les SI développés en no code sont soumises aux mêmes exigences de cybersécurité, notamment à l’obligation d’homologation fondée sur l’analyse de risques (Cf. [RGS](https://cyber.gouv.fr/reglementation/reglementation-identite-confiance-numerique/securite-echanges-voie-electronique/referentiel-general-de-securite/documents-referentiel-general-de-securite/) ANSSI).
* La directive européenne NIS2 renforce les obligations de sécurité applicables aux systèmes publics, en incluant la gestion des risques liés aux fournisseurs et aux plateformes no code. La directive (UE) 2022/2555 « [NIS 2](https://eur-lex.europa.eu/legal-content/FR/TXT/?uri=CELEX:32022L2555) » - comme le règlement IA - imposent une approche fondée sur le risque global du système, et non sur la seule infrastructure technique. Pour illustrer cette application, l’administration doit par exemple évaluer la sécurité de son fournisseur d’IA.
* Le recours au no code n’emporte aucune dérogation au RGPD : l’administration demeure responsable des traitements de données personnelles, quels que soient les outils utilisés. Cela ne veut pas dire auditer les serveurs d'OpenAI ou Anthropic par exemple mais :

1. identifier les risques liés à l’usage du fournisseur (hébergement, chiffrement...etc Cf [OpenAI Trust Portal](https://trust.openai.com/) ou [Anthropic Trust Center](https://trust.anthropic.com/)
2. vérifier que le fournisseur donne des garanties crédibles sur l'usage des données : par exemple peut on désactiver l'enregistrement des prompts pour le réentrainement des modèles, existe-t-il une offre permettant d'isoler le cas d'usage à une organisation...Etc.
3. documenter cette vérification et adapter le niveau d’exigence au niveau de sensibilité des données et des usages (données concernées, usages par les agents, charte IA...Etc).

* Lorsqu’un outil no code intègre de l’IA, il relève du champ du RIA, avec des obligations renforcées pour les systèmes à haut risque, fréquents dans le secteur public. Notamment, développement en no code ne dispense pas des exigences de transparence algorithmique (art 50 RIA).
* Les productions issues du no code peuvent être assimilées à des documents administratifs communicables, et soulèvent là aussi des enjeux de transparence. Si vous prenez une décision envers un usager, une association, une entreprise ou un autre acteur public à l'aide d'un traitement algorithmique, alors vous êtes soumis aux obligations liées à la transparence des algorithmes, prévues par le code des relations entre le public et l'administration (CRPA)
